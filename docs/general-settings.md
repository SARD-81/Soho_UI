# تنظیمات عمومی سامانه

تب «تنظیمات عمومی» نخستین تب صفحه `/settings` است و تنظیمات هویت سامانه، زمان، منطقه زمانی، NTP و ساعت سخت‌افزاری را مدیریت می‌کند.

## APIهای خواندنی

| قابلیت | Method | Endpoint | رفتار UI |
|---|---|---|---|
| اطلاعات زمان سیستم و سخت‌افزار | GET | `/api/system/time/` | نمایش زمان محلی و UTC سیستم‌عامل، زمان محلی و UTC سخت‌افزار، منطقه زمانی و وضعیت NTP |
| فهرست منطقه‌های زمانی | GET | `/api/system/time/zones/` | منبع Autocomplete منطقه زمانی |
| نام میزبان | GET | `/api/system/hostname/` | نمایش و مقدار اولیه فیلد نام میزبان |
| نسخه سامانه | GET | `/api/system/version/` | نمایش محتوای فایل نسخه نصب‌شده |

Queryها فقط هنگام فعال بودن تب عمومی mount می‌شوند. پس از mutation مرتبط، Query همان بخش invalidate و مجدداً از backend خوانده می‌شود.

## قرارداد Responseهای فعلی

### زمان سیستم و سخت‌افزار

Normalizer در درجه اول ساختار واقعی زیر را می‌خواند:

```json
{
  "data": {
    "os_time": {
      "utc": "2026-07-22 11:16:24 UTC",
      "local": "2026-07-22 14:46:25 +0330"
    },
    "hw_time": {
      "utc": "2026-07-22 11:16:28 UTC",
      "local": "2026-07-22 14:46:28 +03:30"
    },
    "timezone": "Asia/Tehran",
    "ntp": {
      "enabled": false,
      "synchronized": false,
      "servers": ["0.debian.pool.ntp.org"]
    }
  }
}
```

Mapping نهایی:

- `data.os_time.local` → زمان محلی سیستم‌عامل
- `data.os_time.utc` → زمان UTC سیستم‌عامل
- `data.hw_time.local` → زمان محلی RTC
- `data.hw_time.utc` → زمان UTC RTC
- `data.timezone` → منطقه زمانی فعال
- `data.ntp.enabled` → وضعیت NTP
- `data.ntp.synchronized` → وضعیت همگام‌سازی
- `data.ntp.servers` → مقدار اولیه فهرست سرورهای NTP

### نام میزبان

`data.hostname` هم به‌عنوان مقدار فعلی و هم مقدار پایدار فرم استفاده می‌شود.

### نسخه سیستم

در Response فعلی، `data.content` اولویت اصلی نمایش نسخه است. اگر موجود نباشد، `data.raw_text` و سپس `data.version` بررسی می‌شوند. مسیر فایل از `data.file_path` نیز در مدل نگهداری می‌شود.

### فهرست منطقه‌های زمانی

اگر `data` آرایه باشد، تمام مقادیر معتبر بدون تکرار استخراج و به ترتیب انگلیسی مرتب می‌شوند.

## تغییر نام میزبان

- Endpoint: `POST /api/system/hostname/set/`
- Body: `{ "hostname": "soho" }`
- قبل از ارسال، نام میزبان مطابق RFC 1123 اعتبارسنجی می‌شود.
- مقدار به lowercase تبدیل می‌شود.
- هر label حداکثر ۶۳ کاراکتر و کل hostname حداکثر ۲۵۳ کاراکتر است.
- قبل از ارسال، مودال تایید اثر تغییر هویت شبکه‌ای سامانه را توضیح می‌دهد.

## تغییر منطقه زمانی

- Endpoint: `POST /api/system/time/set-timezone/`
- Body: `{ "timezone": "Asia/Tehran" }`
- انتخاب فقط از فهرست معتبر دریافت‌شده از backend انجام می‌شود.
- مودال تایید درباره اثر تغییر روی لاگ‌ها و زمان‌بندی سرویس‌ها نمایش داده می‌شود.
- تغییر timezone ساعت UTC را تغییر نمی‌دهد و فقط تفسیر زمان محلی را عوض می‌کند.

## مدیریت NTP

- Endpoint: `POST /api/system/time/ntp/`
- Body: `{ "enabled": true, "servers": ["0.debian.pool.ntp.org"] }`
- سرورهای دریافت‌شده از `data.ntp.servers` مستقیماً فرم را پر می‌کنند.
- کاربر می‌تواند چند سرور اضافه یا حذف کند.
- در حالت فعال، وجود حداقل یک سرور معتبر اجباری است.
- در حالت غیرفعال، آرایه سرورها می‌تواند خالی باشد.
- مقادیر تکراری قبل از ارسال حذف می‌شوند.
- تغییر وضعیت NTP قبل از ارسال نیازمند تایید است.

## تنظیم دستی زمان

- Endpoint: `POST /api/system/time/set-time/`
- Body: `{ "time": "2026-07-20 13:53:01" }`
- ورودی با کنترل `datetime-local` و دقت ثانیه دریافت می‌شود.
- زمان اولیه از `data.os_time.local` پر می‌شود.
- تاریخ‌های نامعتبر یا غیرممکن ارسال نمی‌شوند.
- وقتی NTP در backend فعال است، تنظیم دستی غیرفعال است و کاربر باید ابتدا NTP را خاموش و ثبت کند.
- مودال تایید درباره اثر تغییر زمان روی session، لاگ، TLS و وظایف زمان‌بندی‌شده هشدار می‌دهد.

## مدیریت ساعت سخت‌افزاری RTC

- Endpoint: `POST /api/system/time/hwclock/`

### نمایش RTC

```json
{ "action": "show" }
```

### انتقال زمان سیستم‌عامل به RTC

```json
{ "action": "systohc" }
```

### انتقال RTC به سیستم‌عامل

```json
{ "action": "hctosys", "localtime": false }
```

- `localtime: false`: مقدار RTC به‌عنوان UTC تفسیر می‌شود.
- `localtime: true`: مقدار RTC به‌عنوان Local Time تفسیر می‌شود.
- حالت UTC برای بیشتر سرورهای لینوکسی پیشنهاد می‌شود.
- عملیات‌های تغییردهنده نیازمند تایید هستند.

## اصول UI و RTL

- کل صفحه، تب‌ها، کارت‌ها، هدرها، پیام‌ها و مودال تایید دارای `direction: rtl` و `textAlign: right` هستند.
- ترتیب هدر هر کارت برای زبان فارسی بازطراحی شده و آیکون، عنوان و توضیح به‌صورت یک گروه خوانا نمایش داده می‌شوند.
- مقادیر تمام Inputها، Autocomplete، فهرست منطقه زمانی، NTP و حالت disabled از `--color-text` استفاده می‌کنند.
- label و helper textها از `--color-secondary` استفاده می‌کنند.
- مقادیر فنی مانند Hostname، Timezone، زمان، IP و نام سرور از نظر ترتیب کاراکتر LTR باقی می‌مانند، اما از سمت راست تراز می‌شوند.
- فهرست NTP در دسکتاپ به‌صورت دو ستونه و در موبایل تک‌ستونه نمایش داده می‌شود.
- کارت‌ها، Borderها و Hoverها فقط از متغیرهای طراحی پروژه استفاده می‌کنند و رنگ hardcoded برای متن فرم وجود ندارد.
- هر mutation حالت loading مشترک دارد تا ارسال هم‌زمان عملیات سیستمی رخ ندهد.
- خطاهای backend از response استخراج و در toast نمایش داده می‌شوند.
- اگر یکی از Queryهای اطلاعاتی خطا بخورد، بقیه بخش‌های در دسترس همچنان قابل استفاده می‌مانند.
